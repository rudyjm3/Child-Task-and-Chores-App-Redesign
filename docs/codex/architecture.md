# Architecture

## Tech stack
- **Backend**: Plain PHP (no framework — no Laravel/Symfony/Slim), raw PDO for MySQL/MariaDB, prepared statements throughout. `PDO::ATTR_ERRMODE = ERRMODE_EXCEPTION`.
- **Frontend**: Server-rendered HTML from PHP + vanilla JS (`js/*.js`, no build step, no npm/bundler, no framework). Font Awesome 6.7.2 (CDN) for icons; SortableJS (CDN, routine.php only) for drag-reorder.
- **DB**: MySQL/MariaDB, InnoDB, utf8mb4. No migration framework — see "Schema bootstrap" below.
- **Sessions**: native PHP `$_SESSION` (file-based), no JWT/OAuth.
- **No package manager, no Composer, no build pipeline.** Deploy = copy files to a PHP+MySQL host (originally targeted XAMPP/WAMP for local dev, per `docs/child-app-development-build-outline.md`).
- **Testing**: custom lightweight CLI harness (`tests/run.php` + `tests/lib.php`), no PHPUnit. Each `tests/test_*.php` runs as its own process against a freshly dropped/recreated `child_chore_app` database, exercising the real schema-bootstrap code path.

## Why this stack (design rationale)
- **No framework** was a deliberate incremental-build choice (see `docs/child-app-development-build-outline.md`): the project was built phase-by-phase directly against feature requirements, prioritizing "ship the next phase" over architectural investment. This explains the resulting monolith-per-page structure — each `.php` file accreted its own view + mutation logic over many iterations rather than being designed upfront as an API/frontend split.
- **Every page is both a view and an endpoint** (self-posting forms dispatched on `isset($_POST['sentinel_key'])`, see `docs/codex/routes.md`) rather than a REST API layer, because there was never a client-side app to serve — the "frontend" is the server-rendered HTML itself, progressively enhanced with vanilla JS. `preset_tasks_api.php` is the one deliberate exception, built specifically because the preset-picker modal needed to fetch data without a full page reload.
- **Schema bootstrap instead of migration files**: `includes/functions.php` runs `CREATE TABLE IF NOT EXISTS` / `ALTER TABLE ... ADD COLUMN IF NOT EXISTS` for the *entire* schema on every request (bottom of the file). This makes any environment self-healing (a fresh checkout + empty DB "just works" on first page load) at the cost of the schema being defined imperatively in PHP rather than in versioned `.sql` migration files. The one exception is the `preset_tasks_v1` rename migration (`migratePresetTasksSchema()`), which needed multi-step FK renames/backfills too complex for blind `IF NOT EXISTS` idempotency — that one is tracked in `schema_migrations` and has a dedicated runbook (`docs/preset-tasks-migration.md`) plus standalone CLI scripts (`scripts/migrate_preset_tasks.php`, `scripts/rollback_preset_tasks.php`).
- **Snapshot-on-write for routine/preset data**: `routine_preset_tasks`, `routine_completion_tasks`, and `tasks.preset_task_id` all copy the preset's field values at the moment they're used, rather than joining live to `preset_tasks`. This is deliberate: editing a shared preset template must never silently rewrite a parent's already-scheduled routine or a child's completion history. See `docs/preset-tasks-migration.md` and the "Cross-cutting notes" in `docs/codex/schema.md`.
- **Deleting a referenced preset archives instead of failing**: `deletePresetTask()` checks `presetTaskReferenceCounts()` and soft-archives (`is_active=0`, `archived_at`) instead of hard-deleting when the preset is still in use anywhere (routines, tasks, completion history, overtime logs) — avoids both FK violations and silent data loss.
- **`family_root_id` as the real tenancy boundary**: almost nothing is scoped by `$_SESSION['user_id']` directly. `getFamilyRootId()` resolves any linked account (secondary parent, family member, caregiver, child) to the owning main parent, and most tables are queried `WHERE parent_user_id = family_root_id`. This is the app's multi-tenancy model — one "family" = one main parent + everyone linked to them via `family_links`, all sharing one data namespace.
- **Points vs. Stars are two separate currencies**: points (from `child_points`, adjusted via `child_point_adjustments`) drive reward redemption; stars (from `child_levels`/`child_star_adjustments`, rolling-average based) drive the leveling system (`family_level_settings.stars_per_level`). They are computed and logged independently — don't conflate them when reading task/routine completion code.
- **Autism-friendly design as a first-class requirement** (per `docs/ux-ui-design-system.md` §1 and the original build outline): toggleable sounds/animations, no auto-playing media, predictable/reused card layouts across all task and goal screens, high-contrast status badges that never rely on color alone.

## Key conventions
- **File naming**: root-level `.php` files are literal routes (`task.php` → `/task.php`). No pretty URLs, no `.htaccess` rewriting observed.
- **POST dispatch pattern**: `if ($_SERVER['REQUEST_METHOD'] === 'POST') { if (isset($_POST['action_name'])) {...} elseif (isset($_POST['other_action'])) {...} }`. The sentinel key *is* the route name — e.g. a submit button named `create_task` is what selects that branch, not a URL path or `action` field (routine.php's `action=`-style AJAX handlers, added later, are the one file that broke from this pattern).
- **Mutation → redirect (PRG)**: successful POST handlers generally set a flash message (`$_SESSION['flash_message']` / `$_SESSION['children_flash']`) then `header('Location: ...')` + `exit`, avoiding resubmission-on-refresh. AJAX-style POSTs (routine.php's runtime actions, `preset_tasks_api.php`) instead `echo json_encode(...); exit;`.
- **Auth checks are per-file, not centralized middleware.** Every controller re-checks `$_SESSION['user_id']` (and often a role helper) in its own first ~15 lines. `includes/page_setup.php` centralizes this for the 2 pages that use it (`children.php`, `rewards.php`); the other 9 authenticated pages inline their own guard. See `docs/codex/pages.md` "Convention gap" section — don't assume a page uses `page_setup.php` just because most conceptually should.
- **Role model**: `users.role` is the raw DB value (`main_parent`, `family_member`, `caregiver`, `child`; legacy `'parent'` rows are migrated to `main_parent` at boot). `getEffectiveRole()` resolves a `family_member`'s *actual* granted role via `family_links.role_type` (e.g. a `family_member` linked as `secondary_parent`-equivalent). Always prefer `getEffectiveRole()`/`canCreateContent()`/`canAddEditChild()` over reading `$_SESSION['role']` raw, except `dashboard_child.php`'s guard which intentionally checks the raw session role.
- **Error handling**: PDO exceptions are the primary error surface (`ERRMODE_EXCEPTION`); many DB-bootstrap statements are individually wrapped in `try/catch (PDOException $e) { error_log(...); }` and swallowed (used to make `ADD COLUMN IF NOT EXISTS`-style calls idempotent across DB versions where that syntax isn't fully supported). Application-level errors mostly surface as an `$error` string re-rendered into the existing form, not exceptions/HTTP error pages.
- **No ORM / no query builder**: every query is hand-written SQL via `$db->prepare()->execute([...])`. Table/column names are trusted literals (never interpolated from user input); `IN (...)` lists use dynamically-built placeholder strings (`implode(',', array_fill(...))`).
- **Versioning**: `APP_VERSION` constant in `includes/functions.php` (currently `3.27.0`) is used for CSS/JS cache-busting query strings (`?v=<?php echo APP_VERSION; ?>`) and shown in the footer. Some pages hardcode a *different* literal version string in their own `<script src=...?v=3.26.0>` tags instead of using the constant — this has drifted (e.g. goal.php/login.php/register.php/profile.php link `css/main.css?v=3.28.0` while `APP_VERSION` is `3.27.0`). When bumping the version, grep for hardcoded `?v=` strings, don't assume the constant is the only place it lives.
- **Uploads**: user-uploaded files (avatars, task photo-proof) go to `/uploads/`, validated by size/MIME and (for avatars) GD-resized to 100×100. `/uploads/` should be treated as untrusted-content storage, not directly executable — verify `.htaccess`/webserver config denies PHP execution there before deploying.

## Known issues to know about before touching related code
- **`children.php`'s week-schedule JSON endpoint is broken**: `serveWeekScheduleJson()` (called line 56) and `buildChildWeekSchedule()` (called line 109) are called but **not defined anywhere in the repository** (verified by grep across `includes/functions.php` and the full tree). This will fatal with "Call to undefined function" the moment either code path executes. The equivalent feature works correctly in `dashboard_parent.php` via a local closure `$buildWeekSchedule` (lines 537–766) — that's the reference implementation if this needs fixing. See `docs/codex/routes.md` for exact line numbers.
- **Two high-traffic notification/adjustment tables have no declared FK** (`child_point_adjustments`, `child_star_adjustments`, `child_notifications`, `parent_notifications`, `routine_points_logs` on their `*_user_id`/`routine_id` columns) — these were added later via inline `ensure*Table()` helper calls rather than the main schema-bootstrap block. Functionally fine (app-level scoping is still correct) but means the DB itself won't catch an orphaned row.

## What a new contributor needs to know
1. **Read `docs/codex/routes.md` and `docs/codex/schema.md` before touching any page** — there's no router or ORM to trace through; the sentinel-key POST pattern and the family_root_id tenancy model are the two things that aren't obvious from a single file in isolation.
2. **Local dev needs a MySQL server reachable at `localhost` with an empty-password `root` user** (`includes/db_connect.php` and `tests/lib.php` both hardcode `DB_HOST=localhost, DB_USER=root, DB_PASS=''`). First page load auto-creates the `child_chore_app` database and full schema.
3. **Run `php tests/run.php` before and after DB-touching changes.** Each `test_*.php` drops and recreates the database, so don't point `DB_NAME` at anything you care about.
4. **Preset Task terminology**: "Preset Task" (table `preset_tasks`) is the reusable template; "Task" (table `tasks`) is a concrete assignment to one child (may or may not derive from a preset, snapshotted if so); "Routine" (table `routines`) is a scheduled ordered group of preset tasks for one child, materialized through the `routine_preset_tasks` junction. Don't confuse "preset task" and "task" when reading code or writing docs — they are different tables with different lifecycles.
5. **The monetization/subscription features mentioned in `docs/child-app-development-build-outline.md`** (paid tiers, AI insights, subscriptions table) were never implemented — there is no `subscriptions` table, no billing code, no feature-gating by plan. Treat that doc as historical planning, not current state.
6. **When adding a new page**, follow `includes/page_setup.php` + `includes/html_head.php` (the pattern `children.php`/`rewards.php` use) rather than copying the inline-guard style most existing pages use — it's the intended convention, just not yet applied everywhere.
